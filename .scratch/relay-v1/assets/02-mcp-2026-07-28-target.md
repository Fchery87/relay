# Research: MCP client target — the 2026-07-28 specification

Sources: blog.modelcontextprotocol.io 2026-07-28 release-candidate post (RC locked 2026-05-21, final ships 2026-07-28), SEP-1865 (MCP Apps), SEP-2133 (extensions). Evaluated 2026-07-09.

## What changed (relevant to a client implementer)

- **Stateless core.** `initialize`/`initialized` handshake and `Mcp-Session-Id` removed. Protocol version, client info, capabilities travel in `_meta` on every request; `server/discover` fetches server capabilities on demand. Any request can hit any server instance.
- **Required routing headers.** Streamable HTTP requests must carry `Mcp-Method` and `Mcp-Name`; servers reject header/body mismatches.
- **Multi round-trip requests.** Server-side prompts (elicitation) return `InputRequiredResult` + opaque `requestState`; the client gathers answers and re-issues the call with `inputResponses`. No held-open SSE required. Server-initiated requests only during active client requests.
- **Caching metadata.** List/read results carry `ttlMs` + `cacheScope` — clients may cache `tools/list` accordingly.
- **Tasks is now an extension** (reverse-DNS negotiated): server may answer `tools/call` with a task handle; client drives `tasks/get`/`update`/`cancel`. Server-directed creation; no `tasks/list`.
- **MCP Apps extension**: servers ship sandboxed-iframe HTML UIs, host-rendered, talking JSON-RPC back through the host's audit/consent path.
- **Deprecated core features**: Roots, **Sampling** (replacement: direct LLM provider integration — which Relay already does), Logging (stderr for stdio; OTel for structure).
- **Auth hardening**: RFC 9207 `iss` validation required; DCR declares `application_type` (native, for a daemon with localhost redirect); credentials bound to issuer; refresh-token guidance.
- **Full JSON Schema 2020-12** for tool input/output schemas (composition, `$ref`; never auto-deref external refs; bound depth/validation time). Missing-resource error code moves −32002 → −32602.

## Decision (recommended v1 client surface)

1. **Target `2026-07-28` exclusively.** Final spec lands July 28 — before Relay's MCP ticket will be implemented. Build stateless-first: `_meta`-carried client info, `Mcp-Method`/`Mcp-Name` headers, `server/discover`, `ttlMs`-respecting tools/list cache. No legacy session support in v1 (revisit only if a server the founder needs lags the spec).
2. **Transports: streamable HTTP + stdio.** Both already planned; stdio unchanged for local servers.
3. **Tasks extension: advertise and support as client.** Long-running server work fits Relay's async thread model; task status surfaces as thread events.
4. **Multi round-trip elicitation: map to approval-card UI.** `InputRequiredResult` prompts render like approvals in the browser; answers re-issue the call. Fits the existing awaiting-input thread state.
5. **MCP Apps: NOT supported in v1.** It is not the excluded "live preview" (that meant previewing the user's own app), but rendering third-party server UIs adds an iframe sandbox + consent surface that fights the lightweight bar. Explicitly declining the extension is spec-legal. Revisit post-v1 if a needed server requires it.
6. **Auth: native-app OAuth in the daemon.** DCR with `application_type: native`, localhost redirect, `iss` validation, refresh tokens, credentials bound per issuer — tokens stored in the daemon secrets store, never in Convex. The web UI can deep-link the user into the browser consent flow, but token custody is daemon-only.
7. **Tool schemas:** validate as JSON Schema 2020-12 with bounded depth/time; never fetch external `$ref`s. MCP tool calls flow through the governance chokepoint with risk tiers (already ticketed).
